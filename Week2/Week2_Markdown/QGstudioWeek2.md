## DX11的学习
### 变换
#### 缩放和旋转
* DX11使用的是左手坐标系，OpenGL用的是右手系。在窗口里面xy的坐标是左上角为原点。
* DX11使用的是行向量表示法，OpenGL和Unity里面是列向量表示法，两者有点不一样。导致DX里面是行向量 * 矩阵，其他地方是列向量 * 矩阵
* 在DX的矩阵里面，放缩的元素也是在主对角线，但是旋转的元素放在了左上角。
* 左手系的顺时针旋转=右手系的逆时针旋转
* 反向旋转，用的就是转置的矩阵
#### 平移矩阵
* 向量使用的是(x,y,z,0)，点使用的是(x,y,z,1)，向量平移没用意义，单独的点才有意义。在DX的矩阵里面最下面一行的3*1表示的才是各个坐标的平移量。
#### 复合变换的顺序
矩阵乘法不满足交换律：缩放乘旋转不等于旋转乘缩放
* 先缩放，后旋转，再平移，先旋转再缩放会产生扭曲
#### 四层坐标变换
1. 世界变换，每个物体都会由程序实时生成一个专属于他的世界矩阵（仿射变换矩阵）将物体坐标系放进世界坐标系里面
2. 观察变换，将摄像机放在坐标系的原点的效果，本质上是坐标系变换。由系统构建从世界到观察空间坐标系的变换矩阵。先将世界的原点平移至摄像机，再通过转置矩阵旋转方向。
3. 将观察空间的视野形成的空间几何体映射到一个立方体里面，为后续的裁剪和透视除法做准备。
    * 正交投影：视景体是长方体，通过矩阵映射到齐次裁剪空间立方体
    * 透视投影：视景体就是个四棱台，根据垂直视场角，宽高比，近平面和远平面计算出一个变换矩阵（通过一个函数将深度线性映射到z轴上面）
    * 透视投影用于3D场景，正交投影用于2D场景和UI
4. 在上一步得到的其次裁剪空间里面，xy轴的坐标都是-w~w，z轴的坐标是0~w，所以才叫齐次。经过透视除法，除以w之后才叫做NDC
5. 最后一步，将NDC映射到屏幕空间，将NDC里面的虚拟坐标转化到显示器上面的实际像素位置，同时将深度值提交给深度缓冲区
#### HLSL命名约定
* L local局部空间
* W world世界空间
* V view观察空间
* H honogenous齐次裁剪空间

## DX11的学习

### 前置知识：常量缓冲区的打包方式
1. 基本对齐单位4D向量（16字节，128位）
    * HLSL 常量缓冲区按 16 字节（128 位） 对齐，所有数据最终被组织成若干个 float4 大小的槽
    * C++ 端只需连续提供字节流，HLSL 会按规则将其映射到这些槽中
2. 向量不可拆分
    * HLSL 中声明的 float2、float3 等向量，不会跨两个 4D 向量存储，而是独占一个 4D 向量的前几个分量，剩余分量留空。
3. 相邻变量优先打包进同一 4D 向量
    * 多个标量或小向量会从顶部开始，尽量填满同一个 4D 向量，减少浪费。
4. 结构体内部的成员同样遵循上述打包规则
    * 结构体作为一个整体，其最后一个成员可能会与结构体之后的变量打包到同一个 4D 向量（如果空间允许）
5. 数组的特殊处理
    * 数组的每个元素独立占用一个 4D 向量（即使元素本身很小）
    * 但数组的最后一个元素，如果其后紧跟的变量不是数组也不是结构体，则允许与后续变量打包到同一 4D 向量
### 字节流是什么
字节流就是一段连续的内存数据，没有附带任何类型信息。 字节流就是一段连续的内存数据，没有附带任何类型信息。HLSL 的常量缓冲区（cbuffer）会根据它自己的规则（前面总结的那些打包规则）来解释这些字节，它不知道 C++ 端的结构体定义，只按照 HLSL 声明的布局去读取字节流。

所以如果 C++ 端的结构体布局与 HLSL 的打包规则不一致，数据就会错位。
**那么是谁来管理这个字节流呢？** 这就需要我们在hlsli里面定义常量缓冲区结构体的时候，手动注意内存对齐的问题。或者用pragma pack这个预编译指令和反射工具来确保。

### 光照，常用几何模型，光栅化状态
#### 颜色
在计算机图形学中，我们用一个4维向量 表示颜色：(r, g, b, a)。

在渲染过程中，颜色会经过多次计算（比如光照混合），用浮点数可以保留精度，避免精度损失。最终写入屏幕时会映射到 0~255 的整数。
#### 法向量
光照射到物体表面时，反射的多少取决于光的方向与表面朝向的关系。法向量就是用来描述这个朝向的。

在 3D 建模中，法向量通常由建模软件计算，并保存在模型文件里（每个顶点带一个法向量）。
* 从平面上来的法向量
    * 对于平面，法向量就是垂直于平面的单位向量。
    * 对于曲面上的点，法向量垂直于该点的切平面。
* 顶点上的法向量
    * 对于棱角分明的物体（如立方体），一个顶点可能被多个面共用，而每个面的法向量不同。必须为每个面创建独立的顶点（即使位置相同），这样每个顶点只有一个法向量，才能实现面上棱角分明的光照。例如一个立方体需要24个顶点。
    * 对于平滑表面（如球体），我们可以通过平均相邻面的法向量来得到顶点法向量，或者通过数学公式直接计算。在后面的步骤里面在三角形内插值出每个像素的法向量，产生平滑光照。

#### 光照模型
* ambient环境光：环境光模拟场景中经过多次反射后的均匀光照。它不依赖于光源方向，只是简单地给物体一个基础亮度。

* specular镜面反射光：镜面反射模拟光滑表面的高光。它依赖于观察方向：只有当视线与反射光线方向接近时，才能看到高光。
* 最终颜色：在点光源或聚光灯的情况下，还需要考虑光线的衰减（随距离变暗）和汇聚因子（聚光灯）。最终颜色为：litColor=ambient+diffuse+specular

* 点光源：点光源从一个点向所有方向辐射。亮度随距离衰减，并有一个最大范围。
* 聚光灯：聚光灯从一点沿一个方向照射，形成一个圆锥区域。亮度随角度和距离衰减。

#### 常见的光照模型
1. Blinn-Phong模型   
Phong 的改进版，用“半角向量”替代 Phong 中的反射向量计算，运算更快，高光效果在某些角度更自然。   
    * 优点：
        * 计算量比 Phong 小
        * 高光形状更平滑，尤其在粗糙表面
        * 仍为经验模型，但兼容性好
    * 缺点：
        * 不是物理准确的（能量不守恒）
        * 金属质感表现弱
    * 使用场景:
        * 移动端游戏、低性能平台
        * 需要快速实现高光效果的场景
        * Unity、Unreal 等引擎仍保留作为备选
2. PBR基于物理的渲染（Physically Based Rendering）   
这是现代图形工业的主流标准。核心思想：用微平面理论（Microfacet Theory）和能量守恒来描述材质。
    * 工作流
        * 金属/粗糙度工作流（Metalness-Roughness）
            * 漫反射：用 Lambert 或 Disney Diffuse
            * 镜面反射：用 Cook-Torrance BRDF，包含三个核心项：       
                * 法线分布函数（NDF）：控制高光形状（如 GGX）   
                * 几何函数（Geometry）：控制微平面间的遮挡（如 Smith） 
                * Fresnel 项：描述反射率随角度变化（如 Schlick 近似）
        * 镜面/光泽工作流（Specular-Glossiness）
    * 优点
        * 物理准确，能量守恒，光照效果真实
        * 一套参数（金属度、粗糙度、基础颜色）即可描述几乎所有材质（塑料、金属、陶瓷、皮肤…）
        * 与 IBL（基于图像的光照）结合自然，适合 HDR 环境
    * 缺点
        * 计算量较大（但现代 GPU 已轻松应对）
        * 需要环境贴图（环境光）和准确的光源设置才能发挥效果
    * 使用场景
        * 所有现代 3A 游戏（《荒野大镖客2》《赛博朋克2077》）、电影渲染、Unreal Engine 5、Unity HDRP 等。
3. 各向异性光照模型   
在 PBR 基础上扩展，允许高光沿某个方向拉长，模拟拉丝金属、CD 光盘、头发、丝绸等材质
    * 常见实现：在 Cook-Torrance 中替换法线分布函数为各向异性版本（如 GGX Anisotropic）。
    * 优点：能表现真实世界中方向性高光的材质。
    * 缺点：需要额外的切线方向数据，计算稍重。
4. 次表面散射（SSS, Subsurface Scattering）   
模拟光线进入半透明材质（皮肤、蜡、玉石）后在内部散射再射出的现象。它不是完整的光照模型，而是对漫反射项的增强。
    * 常见实现
        * 预积分皮肤着色（Pre-integrated Skin Shading）
        * 多散射模型（如 Burley SSS）
        * 屏幕空间近似（SSSSS）
    * 优点：使皮肤、牛奶、大理石等材质看起来柔和、真实，不再是“塑料感”。
    * 缺点：计算复杂，通常需要额外的纹理和近似算法。
5. 卡通渲染（Cel Shading / NPR）
    * 常见技术
        * 漫反射用阶梯函数（step）代替连续值，形成明暗色块
        * 高光用阶梯高光或描边高光
        * 外轮廓描边（Backface Outline
        * 定制化的镜面模型

## DX11的学习：XM数学库
### 为什么需要数学库
在 3D 图形程序中，每一帧都要对数以万计的顶点做坐标变换（模型→世界→观察→投影），这些运算涉及大量的向量和矩阵乘法。如果单纯用 C++ 的 float 类型一个个算，性能会非常差。   
**解决方案**：利用CPU的 **SIMD 指令集**（单指令多数据），一条指令同时对多个数据进行运算。
* Intel/AMD CPU 普遍支持 SSE2，它提供 128 位寄存器，可以同时处理 4 个 32 位浮点数（正好对应一个 4D 向量或一个 4×4 矩阵的行/列）。
* DirectXMath 正是基于 SSE2 指令集实现的，让向量/矩阵运算获得硬件加速。
### 运算类型和存储类型
#### 运算时使用的类型：XMVECTOR
`typedef __m128 XMVECTOR;`   
__m128 是一个 128 位的数据类型，内部可看作 4 个 float 的数组   
* **重要特性**：必须 **16 字节对齐**（内存地址末位是 0x0、0x10…），因为 SSE2 指令要求从对齐的内存读取/写入才能达到最高效率   
* 在函数之间传递 XMVECTOR 时，会被直接放入 CPU 寄存器（而不是内存
#### 存储时使用的类型：XMFLOAT2/3/4、XMFLOAT4X4 等 
* 这些结构体是普通的数据容器，没有对齐要求，方便你声明数组、结构体、保存到文件等。它们不能直接用于运算。   
`XMFLOAT3 posStore = {1.0f, 2.0f, 3.0f};   // 存储`   
`XMVECTOR posVec = XMLoadFloat3(&posStore); // 加载到运算向量`
* 加载这些数据类型的时候，不能直接赋值，需要使用对应的存取函数。
* 为什么非要这样？因为 XMVECTOR 类型要求对齐，而存储类型不一定对齐。直接强制转换可能引发异常或性能下降。所以库强制通过函数来“搬运”数据，同时利用 SSE 指令优化。
#### 128位对齐的巧合
XMVECTOR 的 16 字节对齐似乎导致了常量缓冲区按 4D 向量对齐”，其实是 它们共享了相同的设计思想（128 位对齐），并且在工程中常被一起处理，但并不是说 XMVECTOR 对齐直接决定了常量缓冲区的布局。
* XMVECTOR 的 16 字节对齐是 CPU 端 SIMD 优化的要求，属于 DirectXMath 库的实现细节。

* 常量缓冲区按 4D 向量对齐是 GPU 端硬件访问粒度和 HLSL 布局规则的要求，属于 图形管线设计。

* 两者之间 没有直接的因果关系，但都源于 SIMD 效率 的追求，且在实际开发中，为了便于数据映射，我们往往将 C++ 数据结构也设计成 16 字节对齐，从而使得这两个层面的对齐需求在编码实践中统一起来。

### 向量的基本操作
#### 创建向量
1. XMVECTOR v1 = XMVectorSet(1.0f, 2.0f, 3.0f, 4.0f);      // 直接给四个分量
2. XMVECTOR v2 = XMVectorReplicate(5.0f);                   // 所有分量都是5.0f
3. XMVECTOR v3 = XMVectorZero();                            // 全0
4. XMVECTOR v4 = XMVectorSplatX(v1);                        // 取出v1的x，重复四次 -> (1,1,1,1)
#### 获取/修改分量
1. float x = XMVectorGetX(v1);          // 得到分量 x
2. v1 = XMVectorSetX(v1, 10.0f);        // 设置新 x，返回新向量（原向量不变）
#### 算术运算（重载了运算符）
1. XMVECTOR a = XMVectorSet(1,2,3,4);
2. XMVECTOR b = XMVectorSet(5,6,7,8);
3. XMVECTOR c = a + b;      // 分量相加
4. XMVECTOR d = a * b;      // 分量相乘（不是点积）
5. XMVECTOR e = a * 2.0f;   // 标量乘
#### 向量比较（返回掩码，全1或全0）

### 实践要点与常见陷阱
#### 对齐问题
* 永远不要直接对 XMFLOAT* 类型做强制转换为 XMVECTOR 并计算，必须使用 XMLoad* 和 XMStore*。

* 局部变量 XMVECTOR 会自动对齐，但如果你在类成员或全局变量中定义 XMVECTOR，必须用 __declspec(align(16)) 修饰，或者用 XMFLOAT4 存储。
#### 函数调用约定
* 当你写自己的函数并使用 XMVECTOR 参数时，记得遵循 FXMVECTOR、GXMVECTOR 等规则，并加上 XM_CALLCONV，否则在 x86 下可能会因对齐问题崩溃。
#### 矩阵顺序
* *irectXMath 生成的所有矩阵（如 XMMatrixTranslation）都是行主序，与我们之前学的行向量约定一致。
* 矩阵乘法顺序：v * (S * R * T) 等价于 ((v * S) * R) * T，即先缩放、再旋转、最后平移。
* 构建组合矩阵时，用 XMMatrixMultiply 或 * 运算符，注意顺序：M = S * R * T。
#### 性能建议
* 尽量将变换矩阵预先组合好，而不是在每个顶点循环里重复乘多个矩阵。

* 善用常向量和常量，减少不必要的函数调用。
